# AGENTS.md: LIMUZIN GRID MANAGER

Этот файл - краткая рабочая памятка для Codex и других агентных сессий. Он основан на текущем состоянии проекта `v1.0.0` и дополняет, а не заменяет `GRIDBASE.md`, `README.md`, `GITHUB.md`, `roadmap.md` и `versions/GRIDVERSIONS.md`.

## Стек проекта

- Платформа: Windows desktop-приложение для генерации KML-сеток для AlpineQuest.
- Язык: Python `>=3.11,<3.15`.
- GUI: PySide6.
- Координаты: СК-42 / Пулково-42, Гаусс-Крюгер -> WGS84 через `pyproj`.
- Экспорт: KML и ZIP с отдельными `tile_###.kml` для больших квадратов `1000x1000`.
- Управление зависимостями и запуском: `uv`, lock-файл `uv.lock`.
- Сборка EXE: PyInstaller через `build_exe_windows.bat`.
- Тесты: `pytest`.
- Build backend: Hatchling.
- Текущая версия: `1.0.0`, стабильная версия `v1.0.0`.

Основные каталоги:

- `src/limuzin_grid_manager/app/` - прикладные сервисы: экспорт, форматы экспорта, проекты, ресурсы.
- `src/limuzin_grid_manager/core/` - геометрия, CRS, KML, модели, нумерация, статистика и проверки.
- `src/limuzin_grid_manager/ui/` - PySide6-интерфейс и 2D-предпросмотр.
- `tests/` - автотесты ядра, экспорта, проектов, UI-smoke и вкладки экспорта.
- `versions/` - индекс версий и релизные заметки.

## ExecPlans

Для сложных функций, многошаговых изменений, значимых рефакторингов, задач с неизвестными рисками или работ, которые могут занять несколько часов, использовать ExecPlan от проектирования до реализации.

ExecPlan - это живой план выполнения, описанный в `PLANS.md` в корне репозитория. Правильный путь к инструкции из корня проекта:

```text
PLANS.md
```

Перед созданием или исполнением ExecPlan нужно прочитать `PLANS.md` целиком. Если задача не требует ExecPlan, работай по обычным правилам из этого `AGENTS.md`.

## Правила кода

- Не смешивать UI и координатную математику. Новую геометрию, нумерацию, проверки и экспорт добавлять в `core`, а UI должен только собирать параметры и показывать результат.
- Проверки, влияющие на возможность экспорта, держать в `src/limuzin_grid_manager/core/stats.py` или рядом с ним, а не только в UI.
- Соблюдать порядок координат: пользователь вводит `X/Y`, в `pyproj` передается `Y/X`, в KML пишется `lon,lat,0`.
- Логика зоны Гаусса-Крюгера: `zone = int(abs(Y) // 1_000_000)`, EPSG `28400 + zone`, поддерживаемые зоны `1..32`.
- Области, пересекающие границу зон, должны блокироваться с понятной ошибкой. Автоматически разрезать многозонные области нельзя, пока это не станет отдельной задачей.
- Сетка строится сверху вниз и слева направо. Нумерация может быть обычной, змейкой или спиралью в соответствии с настройками.
- Дефолтный KML-стиль должен оставаться совместимым со старым поведением: черные линии, заливка отключена, `<fill>0</fill>`.
- Настраиваемая заливка хранится в `KmlStyle`: для `1000x1000` есть режимы и палитры, для `100x100` только один общий цвет и прозрачность.
- Переименование больших квадратов `1000x1000` не должно менять номера малых квадратов `100x100` внутри них.
- ZIP-экспорт доступен только при включенной сетке `1000x1000`; имена файлов внутри архива остаются `tile_###.kml`.
- `.lgm.json` загружается строго: boolean-поля только JSON `true`/`false`, поврежденные списки имен или цветов должны давать понятную ошибку, а не внутренний traceback.
- Последний открытый или сохраненный `.lgm.json` хранится через локальные `QSettings`, а не в репозитории и не внутри файла проекта.
- Для крупных экспортов сохранять потоковую запись KML, точный прогресс, отмену экспорта, проверку свободного места и атомарную запись через временный файл.
- Новый формат экспорта добавлять через отдельный сервис или модуль, не разращивая `MainWindow`.
- Не возвращать Tkinter-интерфейс из старого `AQ_GridMaker.py`.
- Любая новая опция экспорта или изменение ядра должны получать тесты хотя бы на уровне `core`.
- Не коммитить и не пушить новую версию, не создавать tag или GitHub Release без ручного теста и явного подтверждения пользователя.
- Не добавлять в git временные и пользовательские артефакты: `.venv/`, `build/`, `dist/`, `.pytest_cache/`, `__pycache__/`, `*.spec`, `*.exe`, `*.kml`, `*.kmz`, `*.zip`, `*.lgm.json`, `.env`, ключи и сертификаты.

- `roadmap.md` - история развития до `v1.0` и бэклог идей после `v1.0`.

- `versions/vX.Y.Z.md` - подробные заметки конкретной версии.

Кодовые точки входа:

- `src/limuzin_grid_manager/__main__.py` - запуск GUI.
- `src/limuzin_grid_manager/core/crs.py` - зоны и преобразование СК-42 / Гаусс-Крюгер в WGS84.
- `src/limuzin_grid_manager/core/geometry.py` - нормализация границ, округление и построение сетки.
- `src/limuzin_grid_manager/core/numbering.py` - схемы нумерации `100x100`.
- `src/limuzin_grid_manager/core/kml.py` - потоковая запись KML и ZIP.
- `src/limuzin_grid_manager/core/stats.py` - ошибки, предупреждения, оценки размера и экспортируемости.
- `src/limuzin_grid_manager/core/models.py` - dataclass-модели и enum-режимы.
- `src/limuzin_grid_manager/app/exporter.py` - прикладной экспорт, проверка места, временный файл и атомарная замена.
- `src/limuzin_grid_manager/app/export_formats.py` - варианты экспорта, расширения, пути и сводка.
- `src/limuzin_grid_manager/app/project.py` - сохранение и открытие `.lgm.json`, пресеты.
- `src/limuzin_grid_manager/app/resources.py` - поиск ресурсов в исходниках и PyInstaller bundle.
- `src/limuzin_grid_manager/ui/main_window.py` - основное окно PySide6 и связка UI-сценариев.
- `src/limuzin_grid_manager/ui/preview.py` - 2D-предпросмотр сетки, нумерации и цветов.

Если документы и код расходятся, сначала проверь текущий код и тесты, затем обнови документацию вместе с изменением поведения.

## Инструкции самопроверки

Перед изменениями:

- Прочитать `GRIDBASE.md` и релевантные модули в `src/limuzin_grid_manager`.
- Проверить `git status --short`, чтобы не перепутать свои изменения с уже существующими.
- Если задача касается версии, релиза или GitHub, свериться с `GITHUB.md` и `versions/GRIDVERSIONS.md`.

После обычных изменений кода:

```powershell
uv run --extra dev pytest
uv run --extra dev python -m compileall src tests
```

После UI-изменений:

- Убедиться, что проходят offscreen smoke-тесты из `tests/test_ui.py`.
- Проверить, что предпросмотр и живая проверка не выполняют тяжелые расчеты на каждый символ без задержки.
- Проверить, что крупные сетки не ломают UI и не требуют полной отрисовки всех подписей.

После изменений экспорта или координатной логики:

- Проверить пересечение зон и зоны вне `1..32`.
- Проверить порядок координат `X/Y` -> `Y/X` -> `lon/lat`.
- Проверить KML как XML, количество `Placemark`, ZIP-имена `tile_###.kml`, заливки и пользовательские имена больших квадратов.
- Проверить отмену, временный файл и то, что старый результат не заменяется при ошибке.

Перед передачей новой версии пользователю:

```powershell
uv lock --offline
uv run --offline --extra dev pytest
uv run --offline --extra dev python -m compileall src tests
$env:UV_OFFLINE='1'; .\build_exe_windows.bat
```

Затем проверить:

- `dist/LIMUZIN_GRID_MANAGER.exe` существует.
- Версия в `pyproject.toml`, `src/limuzin_grid_manager/__init__.py` и `version_info.txt` согласована.
- Если менялась версия, обновлены `README.md`, `USER_GUIDE.md` при изменении пользовательских сценариев, `GRIDBASE.md`, `roadmap.md`, `versions/GRIDVERSIONS.md` и `versions/vX.Y.Z.md`.
- Рабочее дерево не содержит лишних build/cache-артефактов.
- EXE не добавлен в git; он прикрепляется к GitHub Release как asset.

Перед публикацией публичного репозитория или релиза дополнительно:

```powershell
git status --short --branch --ignored
git ls-files
git grep -n -i "token\|secret\|password\|api_key"
```

В индексе и истории не должно быть секретов, `.env`, локальных `.lgm.json`, пользовательских KML/KMZ/ZIP-экспортов, EXE, архивов или личных данных.
