# GRIDBASE: LIMUZIN GRID MANAGER

Этот файл — базовая инструкция по проекту LIMUZIN GRID MANAGER. Его можно давать Codex в новых чатах как источник контекста: что делает приложение, как считает сетки, как преобразует координаты, как устроен интерфейс и какие правила нельзя ломать.

## Назначение

LIMUZIN GRID MANAGER — Windows-приложение для генерации KML-сеток для AlpineQuest.

Приложение строит сетку по двум точкам:

- `NW` — верхняя левая точка области.
- `SE` — нижняя правая точка области.
- Формат координат: СК-42 / Пулково-42, Гаусс-Крюгер.
- Единицы: метры.
- `X` — северинг, координата по северу.
- `Y` — восток, координата по востоку.

Поддерживаемые сетки:

- `1000x1000` метров.
- `100x100` метров.
- Одновременно `1000x1000` и `100x100`.
- Только `100x100` на всю область.

Поддерживаемый экспорт:

- Один общий `.kml`.
- `.zip`, где каждый большой квадрат `1000x1000` сохраняется отдельным `tile_###.kml`.
- Один общий `.svg` с метрическим `viewBox`, слоями, подписями и стилями из `KmlStyle`.
- Один общий `.geojson` как `FeatureCollection` с полигонами WGS84 и свойствами ячеек.
- Один общий `.csv` в UTF-8 with BOM с разделителем `;`, границами СК-42 и центрами WGS84.
- Начиная с `v0.6.0`, настройки экспорта вынесены в отдельную вкладку `Экспорт`: формат, папка, имя файла и сводка перед запуском.
- Начиная с `v0.7.0`, рабочее состояние можно сохранять в `.lgm.json`: координаты, настройки сетки, имена квадратов, KML-стиль и настройки экспорта.
- Начиная с `v0.8.0`, экспорт защищен для крупных областей: есть предупреждения и лимит по количеству KML-объектов, оценка размера результата, проверка свободного места, отмена экспорта и атомарная запись через временный файл.
- Начиная с `v0.8.5`, валидация зон и загрузка `.lgm.json` стали строже: зоны Гаусса-Крюгера вне `1..32` блокируются на этапе статистики, boolean-поля проекта читаются только как JSON `true`/`false`, поврежденные списки проекта дают понятную ошибку.

Приложение также запоминает последний открытый или сохраненный `.lgm.json` через `QSettings` и автоматически открывает его при следующем запуске.

Пользовательский многозонный экспорт еще не включен и остается этапом `v1.5.0`.
- Начиная с `v1.5.0`, многозонный экспорт включен для KML, ZIP, SVG, GeoJSON и CSV: writer-ы выбирают трансформер по зоне экспортируемой ячейки, нумерация больших квадратов остается глобальной, ZIP сохраняет `tile_###.kml`, а ячейка, пересекающая границу зоны внутри себя, блокируется понятной ошибкой.
- Начиная с `v1.6.0`, SVG-подписи малых квадратов `100x100` центрируются через SVG baseline `middle`, не меняя KML/ZIP/GeoJSON/CSV и не меняя координатную математику.
- Начиная с `v2.0.0`, возможности этапов `v1.1.0`-`v1.6.0` зафиксированы как стабильный major-релиз: KML/ZIP, SVG, GeoJSON, CSV, темы интерфейса и многозонный экспорт прошли единую релизную проверку и ручной тест пользователя.
- Начиная с `v2.0.1`, в кодовой базе появился отдельный foundation-слой для point-flow: `PointRecord`, `PointStyle`, point-KML writer и point export service. Этот слой пока не встроен в `MainWindow`, не расширяет `GridOptions`, `ExportMode` и `.lgm.json` и служит базой для следующих этапов Excel/point-функциональности.

UI для этого сценария еще не добавлен, но importer и автотесты уже зафиксированы.

Важное правило KML-стиля:

- По умолчанию стиль совместим со старым поведением: черные непрозрачные линии, заливка отключена.
- Начиная с `v0.4.0`, пользователь может настраивать цвет и толщину линий больших и малых квадратов.
- Заливка больших квадратов `1000x1000` поддерживает один цвет, палитру по номерам и пользовательскую палитру.
- Заливка `100x100` поддерживается через один общий цвет и прозрачность для всех малых квадратов; отдельных настроек для каждого малого квадрата нет.
- Если заливка `100x100` включена, каждый малый полигон получает свой `<PolyStyle>` с этим общим цветом.
- Для отключенной заливки используется `<fill>0</fill>`, а не прозрачный общий стиль, чтобы линии оставались видимыми.

## Координатная логика

Входные координаты пользователь вводит как метры в СК-42 / Пулково-42, Гаусс-Крюгер:

- `NW X`, `NW Y`.
- `SE X`, `SE Y`.

В UI допускаются пробелы и запятая:

- `5660000`
- `5 660 000`
- `5660000,0`

Парсинг выполняется в `parse_meter()`:

- Удаляются пробельные символы.
- Запятая заменяется на точку.
- Значение преобразуется в `float`, затем в `int`.

Дальше координаты нормализуются в `normalize_bounds()`:

- `x_top = max(x_nw, x_se)`
- `x_bottom = min(x_nw, x_se)`
- `y_left = min(y_nw, y_se)`
- `y_right = max(y_nw, y_se)`

Это позволяет приложению не зависеть от того, действительно ли пользователь ввел NW выше SE: область все равно приводится к корректным верхней/нижней и левой/правой границам.

Ширина и высота области:

- `width_m = y_right - y_left`
- `height_m = x_top - x_bottom`

## Преобразование СК-42 / Гаусс-Крюгер в WGS84

Файл: `src/limuzin_grid_manager/core/crs.py`.

Логика зоны:

```python
zone = int(abs(y) // 1_000_000)
```

Например:

```text
Y = 6650000 -> zone = 6
```

EPSG строится так:

```python
epsg = 28400 + zone
```

То есть:

- Зона 1 -> EPSG:28401
- Зона 6 -> EPSG:28406
- Зона 32 -> EPSG:28432

Приложение разрешает зоны `1..32`. Для зоны создается трансформер:

```python
crs_src = CRS.from_epsg(28400 + zone)
crs_dst = CRS.from_epsg(4326)
transformer = Transformer.from_crs(crs_src, crs_dst, always_xy=True)
```

Важный порядок координат:

- Вход приложения: `x, y`, где `x` — северинг, `y` — восток.
- Для `pyproj` при `always_xy=True` передается `(y, x)`, то есть `(easting, northing)`.
- На выходе получается `(lon, lat)`.
- В KML координаты пишутся как `lon,lat,0`.

Функция:

```python
lon, lat = transformer.transform(y, x)
```

Многозонные области:

- Если область попадает в несколько зон Гаусса-Крюгера, экспорт разрешается, когда каждая экспортируемая ячейка целиком лежит в одной зоне или касается границы только ребром.
- KML, ZIP, GeoJSON и CSV выбирают `pyproj`-трансформер по зоне конкретной ячейки; SVG не преобразует координаты, но пишет `data-zone` на корне и на прямоугольниках/подписях.
- Если в режиме `none` одна ячейка `1000x1000` или `100x100` пересекает внутреннюю миллионную границу зоны, экспорт блокируется понятной ошибкой с предложением включить округление или выровнять `Y`.
- Если вычисленная зона находится вне поддерживаемого диапазона `1..32`, экспорт блокируется на этапе статистики.
- В `core/zones.py` есть разбиение `Bounds` на `ZoneSegment`: каждый сегмент хранит номер зоны и границы части исходной области.
- `GridStats.zone_segments` хранит подготовленную последовательность сегментов для сводки и определения многозонности.
- Автоматическое разрезание одной ячейки на зональные части пока не подключено; вместо этого блокируется только ячейка, у которой граница зоны проходит внутри полигона.
- Сообщение пользователю для такого случая: включить округление или задать `Y` так, чтобы граница зоны проходила по ребру ячейки.

## Округление границ

Файл: `src/limuzin_grid_manager/core/geometry.py`.

Режимы округления:

- `in` — внутрь / обрезать.
- `out` — наружу / расширить.
- `none` — без округления.

Округление внутрь:

- `x_top` округляется вниз к шагу.
- `x_bottom` округляется вверх к шагу.
- `y_left` округляется вверх к шагу.
- `y_right` округляется вниз к шагу.

Округление наружу:

- `x_top` округляется вверх к шагу.
- `x_bottom` округляется вниз к шагу.
- `y_left` округляется вниз к шагу.
- `y_right` округляется вверх к шагу.

Если после округления площадь стала нулевой или отрицательной, это ошибка:

```text
x_top <= x_bottom или y_right <= y_left
```

Выбор шага первичного округления:

- Если включена только сетка `1000x1000`, шаг округления `1000`.
- Во всех остальных случаях, где есть `100x100`, шаг округления `100`.

Если выбран режим `none`, приложение не расширяет и не обрезает границы. При этом неполные остатки по краям не экспортируются, потому что количество рядов и колонок считается целочисленным делением.

## Построение сетки

Количество рядов и колонок:

```python
rows = bounds.height_m() // step
cols = bounds.width_m() // step
```

Порядок обхода:

- Ряды идут сверху вниз.
- Колонки идут слева направо.
- Для каждой ячейки:
  - `x_top = bounds.x_top - row * step`
  - `y_left = bounds.y_left + col * step`

Углы прямоугольника в СК-42:

```python
[
    (x_top, y_left),       # NW
    (x_bottom, y_left),    # SW
    (x_bottom, y_right),   # SE
    (x_top, y_right),      # NE
    (x_top, y_left),       # close
]
```

Затем каждая точка преобразуется в WGS84 и пишется в KML как `lon,lat,0`.

## Нумерация квадратов

Файлы:

- `src/limuzin_grid_manager/core/geometry.py` — старая змейка для больших квадратов `1000x1000`.
- `src/limuzin_grid_manager/core/numbering.py` — расширяемая нумерация малых квадратов `100x100`.

Змейка для больших квадратов идет по строкам сверху вниз:

- Четная строка `0, 2, 4...` — слева направо.
- Нечетная строка `1, 3, 5...` — справа налево.

Формула:

```python
if row % 2 == 0:
    index = row * cols + col
else:
    index = row * cols + (cols - 1 - col)
```

Пользовательские номера начинаются с `1`, поэтому:

```python
number = index + 1
```

Для больших квадратов `1000x1000` имена файлов и папок используют формат `001`, `002`, `003`.

Для малых квадратов `100x100` внутри каждого большого квадрата нумерация локальная: `1..100`.

Начиная с `v0.3.0`, большие квадраты `1000x1000` можно переименовывать по отдельности:

- модель хранится в `GridOptions.big_tile_names`;
- ключ — стандартный номер большого квадрата: `1`, `2`, `3`, ...;
- значение — пользовательское имя;
- пустые имена отбрасываются и означают стандартное имя;
- переименование большого квадрата не меняет нумерацию малых квадратов `100x100` внутри него;
- имена ZIP-файлов остаются стандартными: `tile_001.kml`, `tile_002.kml`, ...

Начиная с `v0.2.0`, нумерация `100x100` поддерживает:

- обычную нумерацию;
- змейку;
- спираль от центра наружу;
- спираль от края к центру;
- направление по строкам или по колонкам для обычной нумерации и змейки;
- направление спирали: по часовой стрелке или против часовой стрелки;
- стартовый угол `NW`, `NE`, `SW`, `SE` для обычной нумерации и змейки;
- якорь центра `NW`, `NE`, `SW`, `SE` для спирали.

Дефолт совместим с `v0.0.1`: змейка по строкам, старт из `NW`.

Для четной сетки `10x10` у спирали нет одной центральной клетки, поэтому выбранный якорь центра определяет один из четырех центральных квадратов. Дефолтный якорь — `NW`, то есть верхний левый из центральных четырех.

Режим `спираль от края к центру` строится как обратный путь к выбранному центральному якорю: последний номер оказывается в якоре центра.

## KML-экспорт

Файл: `src/limuzin_grid_manager/core/kml.py`.

Есть два основных экспортера:

- `write_kml_all(out_path, bounds, options, progress=None, cancelled=None)`
- `write_zip_per_big_tile(out_zip_path, bounds, options, progress=None, cancelled=None)`

Перед экспортом вызывается `ensure_exportable()`, который проверяет:

- Положительную площадь области.
- Что включена хотя бы одна сетка.
- Что ZIP используется только с включенной сеткой `1000x1000`.
- Что многозонная область не содержит экспортируемую ячейку, пересекающую границу зоны внутри себя.
- Что зона Гаусса-Крюгера входит в поддерживаемый диапазон `1..32`.
- Что после округления есть хотя бы один полный квадрат нужного размера.
  - Что экспорт не превышает лимит примерно `1 000 000` объектов экспорта.

Начиная с `v0.8.0`, статистика дополнительно оценивает:

  - количество объектов экспорта, которые будут записаны;
- примерный размер результата;
- риск слишком большого ZIP по количеству `tile_###.kml`;
- необходимость упрощать предпросмотр для крупных областей.

Прикладной `export_grid()` в `app/exporter.py` пишет во временный файл рядом с целевым путем и заменяет результат только после успешного завершения. Если экспорт отменен или упал с ошибкой, временный файл удаляется, а старый результат не заменяется.

Общий KML:

- Создается один `.kml`.
- Если включена сетка `1000x1000`, каждый большой квадрат пишется в отдельную KML-папку.
- Если для большого квадрата задано пользовательское имя, оно используется как имя KML-папки и имя большого полигона.
- Если включена `100x100`, она пишется вложенной папкой внутри каждого большого квадрата.
- Если `1000x1000` выключена, а `100x100` включена, вся область экспортируется как одна папка `100x100`.

ZIP:

- Создается `.zip`.
- Каждый большой квадрат `1000x1000` становится отдельным файлом:

```text
tile_001.kml
tile_002.kml
tile_003.kml
...
```

- Если включена `100x100`, малые квадраты пишутся внутри соответствующего tile-файла.
- Пользовательское имя большого квадрата меняет имя документа и большого полигона внутри tile-KML, но не меняет имя файла `tile_###.kml`.

  KML пишется потоково в файл, чтобы не держать большие документы полностью в памяти. Для ZIP каждый tile-файл формируется как строка и сразу записывается в архив.

## SVG-экспорт

Файл: `src/limuzin_grid_manager/core/svg.py`.

SVG пишется как обычный XML/text без новой зависимости. Он использует метрический `viewBox` с началом в верхнем левом углу расчетной области: ось SVG `x` соответствует смещению по `Y`, а ось SVG `y` растет вниз от верхней границы `X`. Если включены большие квадраты, viewBox строится по округленным `1000x1000` границам; если включена только сетка `100x100`, viewBox строится по округленным `100x100` границам.

SVG содержит:

- группу `grid-1000` с `data-layer="1000x1000"`, если включены большие квадраты;
- группу `grid-100` с `data-layer="100x100"`, если включены малые квадраты;
- прямоугольники `<rect>` для экспортируемых ячеек;
- подписи `<text>` для номеров и пользовательских имен;
- технические атрибуты `data-layer`, `data-zone`, `data-big-number` и `data-small-number`;
- цвета, толщины линий и заливки из `KmlStyle`.

Подписи малых квадратов `100x100` пишутся в геометрический центр прямоугольника: `x` равен середине SVG-ширины ячейки, `y` равен середине SVG-высоты ячейки, а вертикальная привязка для этого слоя использует `dominant-baseline="middle"` и `alignment-baseline="middle"`. Это правило относится только к SVG-экспорту и не меняет KML-подписи, GeoJSON, CSV или нумерацию.

## GeoJSON-экспорт

Файл: `src/limuzin_grid_manager/core/geojson.py`.

GeoJSON пишется как один JSON-документ без новой зависимости через стандартный модуль `json`. Результат имеет тип `FeatureCollection`. Каждая экспортируемая ячейка становится `Feature` с геометрией `Polygon` в WGS84. Порядок координат внутри GeoJSON такой же, как ожидают GIS-инструменты: `[lon, lat]`.

Свойства feature включают:

- `layer` — `1000x1000` или `100x100`;
- `zone` — зона Гаусса-Крюгера для текущей однозонной области;
- `cell_size_m` — размер ячейки в метрах;
- `big_number` и `big_name` — номер и пользовательское имя большого квадрата, если оно есть;
- `small_number` — номер малого квадрата, если feature относится к `100x100`;
- `x_top`, `x_bottom`, `y_left`, `y_right` — границы ячейки в СК-42.

Писатель не накапливает все features в большом списке: он пишет `FeatureCollection` последовательно и сохраняет проверку отмены через общий `ProgressTracker`.

## CSV-экспорт

Файл: `src/limuzin_grid_manager/core/csv_export.py`.

CSV пишется через стандартный модуль `csv` как один файл в UTF-8 with BOM с разделителем `;`. BOM нужен, чтобы файл удобнее открывался в Excel в русской локали. Каждая экспортируемая ячейка становится отдельной строкой.

Заголовки CSV:

```text
layer;zone;big_number;big_name;small_number;x_top;x_bottom;y_left;y_right;center_lon;center_lat
```

Поля `center_lon` и `center_lat` — центр ячейки в WGS84. Границы `x_top`, `x_bottom`, `y_left`, `y_right` остаются в метрах СК-42, чтобы CSV был пригоден для табличной проверки исходной расчетной сетки.

## Проекты и пресеты

Файл: `src/limuzin_grid_manager/app/project.py`.

Начиная с `v0.7.0`, приложение сохраняет рабочее состояние в `.lgm.json`.

Файл проекта содержит:

- `schema = "limuzin-grid-manager-project"`;
- `schema_version = 1`;
- версию приложения;
- координаты `NW X`, `NW Y`, `SE X`, `SE Y` как строки из UI;
- `GridOptions`, включая имена больших квадратов, `KmlStyle`, старый `export_mode` и новый совместимый `export_format_id`;
- настройки экспорта: папку и имя файла.

Если файл не является поддерживаемым проектом или JSON поврежден, пользователь должен видеть понятную ошибку.

Путь к последнему открытому или сохраненному проекту хранится не в репозитории и не в `.lgm.json`, а в локальных настройках приложения через `QSettings`. При запуске `MainWindow` пытается открыть этот файл автоматически. Если файл удален, перемещен или поврежден, сохраненная ссылка сбрасывается без блокирующего диалога, а приложение остается в состоянии нового проекта.

Начиная с `v0.8.5`, загрузка проекта валидирует типы строже:

- Boolean-поля проекта (`include_1000`, `include_100`, `snake_big`, `small_fill_enabled`) читаются строго как JSON `true`/`false`; строки вроде `"false"` считаются ошибкой файла проекта.
- Списки имен и цветов проекта должны содержать номер квадрата и значение. Поврежденные записи вроде объекта без `number` должны давать понятную ошибку, а не внутренний `KeyError`.

Встроенные пресеты:

- `Режим v0.0.1` — старое поведение: обе сетки, змейка, черные линии, без заливки, общий KML.
- Пресеты нумерации `100x100` — меняют только схему нумерации.
- Пресеты KML/SVG-стиля — меняют только `KmlStyle`.

Пресеты не должны менять координаты пользователя.

## KML/SVG-стиль

Дефолтный стиль границ:

```xml
<LineStyle>
  <color>ff000000</color>
  <width>1</width>
</LineStyle>
<PolyStyle>
  <fill>0</fill>
  <outline>1</outline>
</PolyStyle>
```

Для больших квадратов ширина линии `2`, для малых — `1`.

Цвет KML использует формат `aabbggrr`, поэтому `ff000000` означает полностью непрозрачный черный.

Начиная с `v0.4.0`, стиль хранится в `GridOptions.kml_style` / `KmlStyle`:

- `big_line_color`, `small_line_color` — RGB-цвета линий в формате `#RRGGBB`;
- `big_line_width`, `small_line_width` — толщина линий;
- `big_fill_mode` — режим заливки больших квадратов:
  - `none` — заливка отключена;
  - `single` — один цвет для всех больших квадратов;
  - `by_number` — палитра по номерам больших квадратов;
  - `custom` — пользовательские цвета для отдельных больших квадратов;
- `big_fill_color` — общий цвет заливки или цвет по умолчанию для пользовательской палитры;
- `big_fill_opacity` — прозрачность заливки больших квадратов в процентах `0..100`;
- `custom_big_fill_colors` — цвета отдельных больших квадратов по стандартным номерам `1`, `2`, `3`, ...
- `small_fill_enabled` — включает заливку малых квадратов `100x100`;
- `small_fill_color` — общий цвет заливки `100x100`;
- `small_fill_opacity` — прозрачность заливки `100x100` в процентах `0..100`.

Для заливки больших квадратов KML-цвет строится из RGB и прозрачности как `aabbggrr`.

Если заливка отключена или прозрачность `0%`, в `PolyStyle` пишется `<fill>0</fill>`. Если прозрачность `1..100%`, для большого квадрата пишется `<fill>1</fill>` и `<color>aabbggrr</color>`.

Если включена заливка `100x100`, она пишется в `PolyStyle` каждого малого квадрата. Цвет и прозрачность при этом задаются одной общей настройкой, а не отдельно для каждого малого квадрата.

## Интерфейс

Файлы:

- `src/limuzin_grid_manager/ui/main_window.py`.
- `src/limuzin_grid_manager/ui/preview.py`.

Интерфейс построен на PySide6:

- `MainWindow(QMainWindow)` — основное окно.
- `ExportWorker(QObject)` — экспорт в отдельном worker-потоке, прогресс и отмена.
- `GridPreviewWidget(QWidget)` — 2D-предпросмотр сетки без карты.
- Экспорт выполняется через `QThread`, чтобы окно не зависало на больших сетках.

Размеры окна:

- Стартовый размер: `1180x720`.
- Минимальный размер: `1040x620`.
- Внутри используется `QSplitter`: слева ввод и параметры, справа вкладки предпросмотра, проверки и экспорта.

Верхняя панель проекта:

- Первая строка: `Новый`, `Открыть`, `Сохранить`, `Сохранить как`, текущий файл проекта или состояние `Новый проект`.
- Вторая строка: выпадающий список `Тема`, выпадающий список `Пресеты` и кнопка `Применить пресет`.
- Двухстрочная раскладка нужна, чтобы длинное имя пресета не налезало на кнопку применения.
- Тема меняет только локальное оформление приложения и не сохраняется в `.lgm.json`.
- Те же действия доступны через меню `Проект`, `Пресеты` и `Вид`; пункты меню имеют правый отступ под горячие клавиши.

Левая панель:

- Ввод координат:
  - `NW X`
  - `NW Y`
  - `SE X`
  - `SE Y`
- Настройки сетки:
  - включить `1000x1000`
  - включить `100x100`
- нумерация `1000x1000` змейкой
- схема нумерации `100x100`
- направление по строкам/колонкам для обычной нумерации и змейки
- направление спирали и якорь центра для спиральной нумерации
- режим округления
- `Имена квадратов`:
  - кнопка `Настроить имена 1000x1000`;
  - диалог с таблицей стандартных номеров, пользовательских имен и кнопками `Сброс`.
- `Стиль KML`:
  - краткое резюме цветов, толщин и режима заливки;
  - кнопка `Настроить стиль KML`;
  - диалог с настройками линий, заливкой `100x100`, режимом заливки `1000x1000`, прозрачностью, палитрой по номерам и пользовательской таблицей цветов больших квадратов.
Правая панель:

- `Предпросмотр`:
  - 2D-схема без карты;
  - большие квадраты `1000x1000`;
  - выбранный большой квадрат;
  - малые квадраты `100x100` внутри выбранного большого квадрата;
  - номера квадратов;
  - текущие цвета линий и заливки из `KmlStyle`;
  - кнопки `Вписать всё`, `К выбранному`, `+`, `-`;
  - масштабирование колесом мыши;
  - перемещение перетаскиванием;
  - выбор большого квадрата кликом.
- `Проверка`:
  - живая проверка ввода;
  - размер области до и после округления;
  - зона Гаусса-Крюгера;
  - количество квадратов;
  - предупреждения;
  - ошибки;
  - краткое резюме текущего KML-стиля.
- `Экспорт`:
  - выбор варианта экспорта: общий KML, ZIP по квадратам `1000x1000`, SVG-схема, GeoJSON или CSV;
  - выбор папки и имени файла;
  - кнопка выбора готового пути файла через системный диалог;
  - сводка перед запуском: сколько файлов будет создано, сколько объектов выбранного формата будет записано, примерный размер результата и что попадет в ZIP, SVG, GeoJSON или CSV;
  - сводка зон экспорта; для многозонной области указывается, что трансформер выбирается по ячейке;
  - вкладка прокручивается через `QScrollArea`, отключает горизонтальную прокрутку и растягивает блоки по доступной ширине, чтобы список форматов не разваливал окно.

Нижняя панель:

- `Сгенерировать`
- `Отменить` во время экспорта
- `Открыть папку`
- progress bar во время экспорта

Живая проверка и предпросмотр работают через `QTimer` с небольшой задержкой, чтобы не пересчитывать статистику на каждый символ без паузы.

Темы интерфейса:

- `system/light` — светлая тема по умолчанию, близкая к прежнему виду;
- `dark` — темная тема для слабого освещения;
- `high-contrast` — контрастная тема для яркого света и слабых экранов;
- тема хранится в `QSettings` по ключу `ui/theme` и не входит в `.lgm.json`;
- `GridPreviewWidget` получает палитру темы, но остается 2D-схемой без картографической подложки.

Предпросмотр не должен ломать интерфейс на больших сетках:

- виджет рисует только видимую часть схемы;
- плотные линии прореживаются;
- подписи скрываются, если они становятся нечитаемыми;
- спиральные номера для очень больших цельных `100x100`-сеток не рассчитываются в предпросмотре, если это потребует построения слишком большой матрицы;
- полный экспорт при этом продолжает работать по прежним правилам.

## Ошибки и предупреждения

Экспорт блокируется при ошибках:

- Площадь области <= 0.
- Выключены обе сетки.
- ZIP выбран без сетки `1000x1000`.
- После округления область исчезла.
- Экспортируемая ячейка пересекает границу зон Гаусса-Крюгера внутри себя.
- Зона Гаусса-Крюгера находится вне поддерживаемого диапазона `1..32`.
- Нет полного квадрата нужного размера для экспорта.
- Экспорт слишком большой: примерно больше `1 000 000` объектов экспорта.

Предупреждения не блокируют экспорт:

- В режиме `none`, если границы не кратны `1000` или `100`, приложение предупреждает, что неполные остатки по краям не попадут в экспорт.
- Если экспорт содержит больше примерно `100 000` объектов, приложение предупреждает о долгой генерации и тяжелом файле.
- Если ZIP содержит очень много `tile_###.kml`, приложение предупреждает о большом архиве.
- Если область слишком велика для подробной отрисовки, предпросмотр упрощается: часть линий и подписей скрывается.

## Тестовая база

Покрытые сценарии:

- `infer_gk_zone(6650000) == 6`.
- Округление `in`, `out`, `none`.
- Ошибка, когда область исчезает после округления.
- Нумерация змейкой.
- Нумерация `100x100`: обычная, змейка, спираль от центра наружу, спираль от края к центру.
- Направление спирали по часовой стрелке и против часовой стрелки.
- Центральные якоря `NW`, `NE`, `SW`, `SE` для сетки `10x10`.
- Отсутствие дубликатов и пропущенных номеров в `10x10`.
- Порядок углов прямоугольника `NW, SW, SE, NE, NW`.
- KML парсится как XML.
- KML содержит ожидаемое количество `Placemark`.
- Point-KML парсится как XML и содержит `Document/Waypoints`, `Placemark`, `Point`, `description`, `ExtendedData/comment` и общий `IconStyle/color`.
- Point-domain нормализует дату в `dd.mm.yyyy`, координаты `X/Y` и цвет точки в формат KML `aabbggrr`.
- Point-import через `openpyxl` читает первый лист с данными, требует заголовок `ФИО | Дата | Координаты`, собирает ошибки по строкам и блокирует экспорт при частично невалидном workbook.
- Дефолтный KML не содержит цветной заливки.
- KML поддерживает цвет и толщину линий больших и малых квадратов.
- Заливка больших квадратов поддерживает один цвет, палитру по номерам, пользовательскую палитру и alpha-канал.
- Заливка `100x100` пишется в каждый малый квадрат одним общим цветом и прозрачностью.
- SVG парсится как XML, содержит слои `1000x1000`/`100x100`, подписи, `data-*` атрибуты, цвета и прозрачность из `KmlStyle`; подписи `100x100` находятся в центре своих прямоугольников.
- GeoJSON парсится как JSON `FeatureCollection`, содержит `Polygon` в WGS84 и свойства `layer`, `zone`, `big_number`, `big_name`, `small_number`.
- CSV содержит UTF-8 BOM, разделитель `;`, заголовки, строки для больших и малых квадратов, границы СК-42 и центры WGS84.
- ZIP содержит `tile_001.kml`, `tile_002.kml`, ...
- Пользовательское имя большого квадрата попадает в KML-папку и имя большого полигона.
- ZIP сохраняет стандартное имя файла `tile_###.kml` при переименовании большого квадрата.
- Переименование большого квадрата не меняет нумерацию `100x100` внутри него.
- Один tile с включенной `100x100` содержит большой полигон и 100 малых.
- Многозонный экспорт проходит для KML/ZIP/SVG/GeoJSON/CSV, если каждая экспортируемая ячейка лежит в одной зоне.
- Смещенная ячейка, пересекающая границу зон Гаусса-Крюгера внутри себя, вызывает ошибку.
- Зоны Гаусса-Крюгера вне `1..32` блокируются на этапе статистики до запуска экспорта.
- Сводка вкладки `Экспорт` показывает количество создаваемых KML/ZIP/SVG/GeoJSON/CSV-файлов.
- `.lgm.json` сохраняется и открывается без потери координат, опций, имен квадратов, палитры и экспорта.
- Последний открытый или сохраненный `.lgm.json` запоминается через `QSettings` и автоматически открывается при следующем запуске.
- Тема интерфейса сохраняется через `QSettings`, восстанавливается при следующем запуске и не сохраняется в `.lgm.json`.
- Неподдерживаемая схема проекта дает понятную ошибку.
- Загрузка `.lgm.json` покрыта проверками строгих boolean-полей и поврежденных списков имен/цветов.
- Большие экспорты получают предупреждения и блокируются при чрезмерном количестве объектов экспорта.
- Прогресс экспорта считает большие и малые полигоны отдельно.
- Отмена экспорта удаляет временный незавершенный файл.
- Offscreen smoke-тесты проверяют создание `MainWindow`, вкладку `Экспорт`, панель проекта, кнопки отмены, применение пресета и автоматическое восстановление последнего проекта.

Перед важными изменениями запускать:

```powershell
uv run --extra dev pytest
uv run --extra dev python -m compileall src tests
```

## Правила для будущей разработки

- Не смешивать UI и координатную математику.
- Новую геометрию и экспорт добавлять в `core`, а UI должен только собирать параметры и показывать результат.
- Все проверки, влияющие на возможность экспорта, держать в `core/stats.py` или рядом с ним, а не только в UI.
- Для больших экспортов сохранять потоковую запись KML.
- Не возвращать Tkinter-интерфейс из старого `AQ_GridMaker.py`.
- Цветная заливка должна оставаться управляемой настройкой `KmlStyle`: для `1000x1000` доступны режимы и палитры, для `100x100` один общий цвет и прозрачность.
- Многозонный экспорт должен оставаться zone-aware: нельзя молча преобразовывать часть ячейки через неверную зону, а смещенную ячейку на границе зоны нужно блокировать до отдельной задачи по разрезанию таких ячеек.
- Предпросмотр карты или схемы держать отдельной вкладкой/панелью, не ломая существующий поток ввода и экспорта.
- Если добавляется новый формат экспорта, он должен идти через отдельный сервис/модуль, а не через разрастание `MainWindow`.
- Point-flow должен развиваться отдельно от grid-flow: новые point-сущности, writer-ы и сервисы не должны расширять `GridOptions`, `ExportMode`, текущую вкладку `Экспорт` и `.lgm.json`, пока это не станет отдельной сознательной задачей.

## Короткая памятка для Codex в новых чатах

Если пользователь просит менять LIMUZIN GRID MANAGER:

1. Прочитай этот `GRIDBASE.md`.
2. Сверь текущие файлы в `src/limuzin_grid_manager`.
3. Координаты: вход `X/Y`, в `pyproj` передавать `Y/X`, в KML писать `lon/lat`.
4. Зона: `int(abs(Y) // 1_000_000)`, EPSG `28400 + zone`; поддерживаемый диапазон зон `1..32`, все остальное должно давать ошибку до экспорта.
   Многозонность находится в `core/zones.py` и writer-ах KML/ZIP/SVG/GeoJSON/CSV: трансформер выбирается по зоне экспортируемой ячейки, а ячейка, пересекающая границу зоны внутри себя, блокируется.
5. Сетка строится сверху вниз, слева направо, номера могут идти обычным порядком, змейкой или спиралью.
6. Дефолтный KML без заливки: линия `ff000000`, `<fill>0</fill>`; настраиваемая заливка `1000x1000` и `100x100` хранится в `KmlStyle`.
7. Переименование больших квадратов не должно менять номера малых `100x100`; ZIP-файлы остаются `tile_###.kml`.
8. ZIP только по большим квадратам `1000x1000`.
9. `.lgm.json` должен загружаться строго: boolean только JSON `true`/`false`, поврежденные списки имен/цветов дают понятную ошибку.
10. Последний открытый или сохраненный `.lgm.json` запоминается в локальных настройках приложения через `QSettings`, а не в репозитории и не внутри самого проекта.
11. Для крупных областей сохраняй предупреждения/лимиты из `core/stats.py`, отмену экспорта и атомарную запись через временный файл.
12. После изменений запускать тесты.
