# GRIDBASE: LIMUZIN GRID MANAGER

Этот файл — базовая инструкция по проекту LIMUZIN GRID MANAGER. Его можно давать Codex в новых чатах как источник контекста: что делает приложение, как считает сетки, как преобразует координаты, как устроен интерфейс и какие правила нельзя ломать.

## Назначение

LIMUZIN GRID MANAGER — Windows-приложение для генерации KML-сеток для AlpineQuest.

Приложение строит сетку по двум точкам:

- `NW` — верхняя левая точка области.
- `SE` — нижняя правая точка области.
- Формат координат: СК-42 / Пулково-42, Гаусс-Крюгер.
- Единицы: метры.
- `X` — северинг, координата по северу.
- `Y` — восток, координата по востоку.

Поддерживаемые сетки:

- `1000x1000` метров.
- `100x100` метров.
- Одновременно `1000x1000` и `100x100`.
- Только `100x100` на всю область.

Поддерживаемый экспорт:

- Один общий `.kml`.
- `.zip`, где каждый большой квадрат `1000x1000` сохраняется отдельным `tile_###.kml`.

Важное правило KML-стиля:

- Цветная заливка отключена.
- Полупрозрачные полигоны не используются.
- Границы квадратов рисуются стандартной непрозрачной черной линией `ff000000`.
- В `PolyStyle` используется `<fill>0</fill>`, то есть заливки нет.
- Не возвращать старую функцию цветной заливки и не добавлять UI-опцию “заливка разными цветами”, если пользователь прямо не попросит изменить это правило.

## Координатная логика

Входные координаты пользователь вводит как метры в СК-42 / Пулково-42, Гаусс-Крюгер:

- `NW X`, `NW Y`.
- `SE X`, `SE Y`.

В UI допускаются пробелы и запятая:

- `5660000`
- `5 660 000`
- `5660000,0`

Парсинг выполняется в `parse_meter()`:

- Удаляются пробельные символы.
- Запятая заменяется на точку.
- Значение преобразуется в `float`, затем в `int`.

Дальше координаты нормализуются в `normalize_bounds()`:

- `x_top = max(x_nw, x_se)`
- `x_bottom = min(x_nw, x_se)`
- `y_left = min(y_nw, y_se)`
- `y_right = max(y_nw, y_se)`

Это позволяет приложению не зависеть от того, действительно ли пользователь ввел NW выше SE: область все равно приводится к корректным верхней/нижней и левой/правой границам.

Ширина и высота области:

- `width_m = y_right - y_left`
- `height_m = x_top - x_bottom`

## Преобразование СК-42 / Гаусс-Крюгер в WGS84

Файл: `src/limuzin_grid_manager/core/crs.py`.

Логика зоны:

```python
zone = int(abs(y) // 1_000_000)
```

Например:

```text
Y = 6650000 -> zone = 6
```

EPSG строится так:

```python
epsg = 28400 + zone
```

То есть:

- Зона 1 -> EPSG:28401
- Зона 6 -> EPSG:28406
- Зона 32 -> EPSG:28432

Приложение разрешает зоны `1..32`. Для зоны создается трансформер:

```python
crs_src = CRS.from_epsg(28400 + zone)
crs_dst = CRS.from_epsg(4326)
transformer = Transformer.from_crs(crs_src, crs_dst, always_xy=True)
```

Важный порядок координат:

- Вход приложения: `x, y`, где `x` — северинг, `y` — восток.
- Для `pyproj` при `always_xy=True` передается `(y, x)`, то есть `(easting, northing)`.
- На выходе получается `(lon, lat)`.
- В KML координаты пишутся как `lon,lat,0`.

Функция:

```python
lon, lat = transformer.transform(y, x)
```

Многозонные области:

- Если левая и правая границы области попадают в разные зоны Гаусса-Крюгера, экспорт блокируется.
- Автоматического разрезания по зонам сейчас нет.
- Сообщение пользователю: область нужно разбить по зонам.

## Округление границ

Файл: `src/limuzin_grid_manager/core/geometry.py`.

Режимы округления:

- `in` — внутрь / обрезать.
- `out` — наружу / расширить.
- `none` — без округления.

Округление внутрь:

- `x_top` округляется вниз к шагу.
- `x_bottom` округляется вверх к шагу.
- `y_left` округляется вверх к шагу.
- `y_right` округляется вниз к шагу.

Округление наружу:

- `x_top` округляется вверх к шагу.
- `x_bottom` округляется вниз к шагу.
- `y_left` округляется вниз к шагу.
- `y_right` округляется вверх к шагу.

Если после округления площадь стала нулевой или отрицательной, это ошибка:

```text
x_top <= x_bottom или y_right <= y_left
```

Выбор шага первичного округления:

- Если включена только сетка `1000x1000`, шаг округления `1000`.
- Во всех остальных случаях, где есть `100x100`, шаг округления `100`.

Если выбран режим `none`, приложение не расширяет и не обрезает границы. При этом неполные остатки по краям не экспортируются, потому что количество рядов и колонок считается целочисленным делением.

## Построение сетки

Количество рядов и колонок:

```python
rows = bounds.height_m() // step
cols = bounds.width_m() // step
```

Порядок обхода:

- Ряды идут сверху вниз.
- Колонки идут слева направо.
- Для каждой ячейки:
  - `x_top = bounds.x_top - row * step`
  - `y_left = bounds.y_left + col * step`

Углы прямоугольника в СК-42:

```python
[
    (x_top, y_left),       # NW
    (x_bottom, y_left),    # SW
    (x_bottom, y_right),   # SE
    (x_top, y_right),      # NE
    (x_top, y_left),       # close
]
```

Затем каждая точка преобразуется в WGS84 и пишется в KML как `lon,lat,0`.

## Нумерация змейкой

Файл: `src/limuzin_grid_manager/core/geometry.py`.

Змейка идет по строкам сверху вниз:

- Четная строка `0, 2, 4...` — слева направо.
- Нечетная строка `1, 3, 5...` — справа налево.

Формула:

```python
if row % 2 == 0:
    index = row * cols + col
else:
    index = row * cols + (cols - 1 - col)
```

Пользовательские номера начинаются с `1`, поэтому:

```python
number = index + 1
```

Для больших квадратов `1000x1000` имена файлов и папок используют формат `001`, `002`, `003`.

Для малых квадратов `100x100` внутри каждого большого квадрата нумерация локальная: `1..100`.

Начиная с `v0.1.0`, нумерация `100x100` задается тремя настройками:

- схема: обычная или змейка;
- направление: по строкам или по колонкам;
- стартовый угол: `NW`, `NE`, `SW`, `SE`.

Дефолт совместим с `v0.0.1`: змейка по строкам, старт из `NW`.

## KML-экспорт

Файл: `src/limuzin_grid_manager/core/kml.py`.

Есть два основных экспортера:

- `write_kml_all(out_path, bounds, options, progress=None)`
- `write_zip_per_big_tile(out_zip_path, bounds, options, progress=None)`

Перед экспортом вызывается `ensure_exportable()`, который проверяет:

- Положительную площадь области.
- Что включена хотя бы одна сетка.
- Что ZIP используется только с включенной сеткой `1000x1000`.
- Что область не пересекает границу зон Гаусса-Крюгера.
- Что после округления есть хотя бы один полный квадрат нужного размера.

Общий KML:

- Создается один `.kml`.
- Если включена сетка `1000x1000`, каждый большой квадрат пишется в отдельную KML-папку.
- Если включена `100x100`, она пишется вложенной папкой внутри каждого большого квадрата.
- Если `1000x1000` выключена, а `100x100` включена, вся область экспортируется как одна папка `100x100`.

ZIP:

- Создается `.zip`.
- Каждый большой квадрат `1000x1000` становится отдельным файлом:

```text
tile_001.kml
tile_002.kml
tile_003.kml
...
```

- Если включена `100x100`, малые квадраты пишутся внутри соответствующего tile-файла.

KML пишется потоково в файл, чтобы не держать большие документы полностью в памяти. Для ZIP каждый tile-файл формируется как строка и сразу записывается в архив.

## KML-стиль

Текущий стиль границ:

```xml
<LineStyle>
  <color>ff000000</color>
  <width>1</width>
</LineStyle>
<PolyStyle>
  <fill>0</fill>
  <outline>1</outline>
</PolyStyle>
```

Для больших квадратов ширина линии `2`, для малых — `1`.

Цвет KML использует формат `aabbggrr`, поэтому `ff000000` означает полностью непрозрачный черный.

Не использовать `alpha=00` для “прозрачности 0”, потому что в KML это делает объект невидимым. Правильная текущая трактовка пользовательского требования: заливка отключена через `<fill>0</fill>`, линия остается видимой и непрозрачной.

## Интерфейс

Файл: `src/limuzin_grid_manager/ui/main_window.py`.

Интерфейс построен на PySide6:

- `MainWindow(QMainWindow)` — основное окно.
- `ExportWorker(QObject)` — экспорт в отдельном worker-потоке.
- Экспорт выполняется через `QThread`, чтобы окно не зависало на больших сетках.

Размеры окна:

- Стартовый размер: `1060x680`.
- Минимальный размер: `920x560`.
- Внутри используется `QSplitter`: слева ввод и параметры, справа проверка и статистика.

Левая панель:

- Ввод координат:
  - `NW X`
  - `NW Y`
  - `SE X`
  - `SE Y`
- Настройки сетки:
  - включить `1000x1000`
  - включить `100x100`
- нумерация `1000x1000` змейкой
- схема, направление и стартовый угол нумерации `100x100`
  - режим округления
- Настройки экспорта:
  - один общий `.kml`
  - `.zip` по квадратам `1000x1000`

Правая панель:

- Живая проверка ввода.
- Размер области до и после округления.
- Зона Гаусса-Крюгера.
- Количество квадратов.
- Предупреждения.
- Ошибки.
- Напоминание: KML-стиль без заливки.

Нижняя панель:

- `Проверить`
- `Сгенерировать`
- `Открыть папку`
- progress bar во время экспорта

Живая проверка работает через `QTimer` с небольшой задержкой, чтобы не пересчитывать статистику на каждый символ без паузы.

## Ошибки и предупреждения

Экспорт блокируется при ошибках:

- Площадь области <= 0.
- Выключены обе сетки.
- ZIP выбран без сетки `1000x1000`.
- После округления область исчезла.
- Область пересекает границу зон Гаусса-Крюгера.
- Нет полного квадрата нужного размера для экспорта.

Предупреждения не блокируют экспорт:

- В режиме `none`, если границы не кратны `1000` или `100`, приложение предупреждает, что неполные остатки по краям не попадут в экспорт.

## Тестовая база

Покрытые сценарии:

- `infer_gk_zone(6650000) == 6`.
- Округление `in`, `out`, `none`.
- Ошибка, когда область исчезает после округления.
- Нумерация змейкой.
- Порядок углов прямоугольника `NW, SW, SE, NE, NW`.
- KML парсится как XML.
- KML содержит ожидаемое количество `Placemark`.
- KML не содержит цветной заливки.
- ZIP содержит `tile_001.kml`, `tile_002.kml`, ...
- Один tile с включенной `100x100` содержит большой полигон и 100 малых.
- Пересечение зон Гаусса-Крюгера вызывает ошибку.

Перед важными изменениями запускать:

```powershell
uv run --extra dev pytest
uv run --extra dev python -m compileall src tests
```

## Правила для будущей разработки

- Не смешивать UI и координатную математику.
- Новую геометрию и экспорт добавлять в `core`, а UI должен только собирать параметры и показывать результат.
- Все проверки, влияющие на возможность экспорта, держать в `core/stats.py` или рядом с ним, а не только в UI.
- Для больших экспортов сохранять потоковую запись KML.
- Не возвращать Tkinter-интерфейс из старого `AQ_GridMaker.py`.
- Не возвращать цветную заливку без отдельного явного решения пользователя.
- Не делать автоматическое разбиение многозонных областей, пока это не станет отдельной задачей.
- Если появится предпросмотр карты или схемы, делать его отдельной вкладкой/панелью, не ломая существующий поток ввода и экспорта.
- Если добавляется новый формат экспорта, он должен идти через отдельный сервис/модуль, а не через разрастание `MainWindow`.
