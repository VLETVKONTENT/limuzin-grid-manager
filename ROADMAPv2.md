# ROADMAPv2: LIMUZIN GRID MANAGER v1.0 -> v2.0

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

В корне репозитория есть `PLANS.md`. Этот файл нужно поддерживать по правилам `PLANS.md`, но его роль - roadmap-уровень для перехода от стабильной версии `v1.0.0` к `v2.0.0`, а не одноразовый план реализации одной функции.

## Purpose / Big Picture

Цель `v2.0` - превратить LIMUZIN GRID MANAGER из стабильного генератора KML-сеток в более универсальный инструмент подготовки сеток для полевой, GIS и векторной работы, не перегружая текущий PySide6-интерфейс.

После выполнения этой дорожной карты пользователь сможет экспортировать ту же расчетную сетку не только в KML/ZIP для AlpineQuest, но и в SVG для векторного редактирования, GeoJSON для GIS-инструментов и CSV для табличной проверки номеров, зон и координат. Приложение также сможет автоматически обрабатывать область, пересекающую несколько зон СК-42 / Пулково-42, Гаусса-Крюгера, разбивая экспорт по зонам внутри одного пользовательского действия.

В `v2.0` также должны появиться темы интерфейса и умеренная оптимизация структуры приложения. При этом map-backed preview, то есть предпросмотр с картографической подложкой, намеренно не входит в `v2.0`: текущий 2D-предпросмотр без карты сохраняется и развивается только настолько, насколько нужно для новых экспортов и многозонности.

## Decision Log

- Decision: `ROADMAPv2.md` оформляется как ExecPlan-style roadmap, а не как короткий список задач.
  Rationale: Переход к `v2.0` затрагивает архитектуру, экспорт, интерфейс и координатную логику; это многошаговая работа, для которой правила `PLANS.md` полезнее краткого бэклога.
  Date/Author: 2026-04-07, Codex + user request.

- Decision: Поддержка разных зон в `v2.0` означает автоматическое разбиение одной прямоугольной области по зонам Гаусса-Крюгера.
  Rationale: Пользователь выбрал вариант `Авторазбиение`, а не ручной выбор зоны и не модель нескольких независимых участков.
  Date/Author: 2026-04-07, user choice.

- Decision: В `v2.0` входят новые форматы SVG, GeoJSON и CSV; DXF не входит в обязательный объем `v2.0`.
  Rationale: Пользователь выбрал `SVG + GeoJSON/CSV`; DXF полезен для CAD-сценариев, но потребует отдельной проверки координат, формата и зависимостей.
  Date/Author: 2026-04-07, user choice.

- Decision: Карта в предпросмотре, включая map-backed preview, остается за пределами `v2.0`.
  Rationale: Пользователь явно сказал, что карту в предпросмотре сейчас не рассматриваем и оставляем для будущих версий.
  Date/Author: 2026-04-07, user instruction.

- Decision: Выбор темы приложения должен храниться в локальных настройках через `QSettings`, а не в переносимом `.lgm.json`.
  Rationale: Тема - предпочтение конкретного рабочего места и экрана, а не часть геометрии, экспорта или проектных данных.
  Date/Author: 2026-04-07, Codex.

- Decision: Для `v1.1.0` новый `format_id` будет жить в прикладном реестре `app/export_formats.py` и сохраняться в `.lgm.json` как дополнительное поле рядом со старым `export_mode`, а `GridOptions.export_mode` останется совместимым внутренним режимом выполнения.
  Rationale: Это дает стабильный идентификатор формата для UI и проекта, но не ломает существующий core-контракт и старые проекты с `export_mode: "kml"` или `export_mode: "zip"`. Будущие этапы смогут расширять реестр без срочной миграции всех core-моделей.
  Date/Author: 2026-04-08, Codex.

- Decision: Для `v1.2.0` GeoJSON и CSV будут отдельными `ExportMode` и отдельными core-writer-модулями, а не режимами внутри KML/SVG writer-ов.
  Rationale: Такой путь сохраняет существующий реестр форматов, не разращивает `MainWindow`, позволяет `.lgm.json` хранить стабильный `export_format_id` для каждого результата и оставляет будущую многозонность отдельным этапом `v1.5.0`.
  Date/Author: 2026-04-08, Codex.

- Decision: GeoJSON и CSV будут писаться потоково через стандартные модули `json` и `csv`, без новых зависимостей.
  Rationale: GeoJSON `FeatureCollection` можно безопасно сформировать как последовательность features с запятыми между ними, не накапливая крупный список в памяти; CSV естественно пишется построчно и должен получить UTF-8 with BOM для Excel в русской локали.
  Date/Author: 2026-04-08, Codex.

- Decision: Для `v1.3.0` темы вынесены в `src/limuzin_grid_manager/ui/themes.py` как QSS + палитра предпросмотра, а не как поля `.lgm.json` или часть core-моделей.
  Rationale: Тема влияет только на локальное отображение интерфейса, а `GridPreviewWidget` рисует часть UI вручную через `QPainter`, поэтому ему нужен маленький UI-уровневый набор цветов без изменения экспортной и проектной схемы.
  Date/Author: 2026-04-08, Codex.

- Decision: Для `v1.3.0` вкладка `Экспорт` оптимизирована внутри `MainWindow`, без вынесения отдельного `ui/export_tab.py`.
  Rationale: Требование этапа - умеренная оптимизация без большого редизайна. После `v1.2.0` вкладка уже была изолирована методом `_build_export_tab()`, поэтому достаточно убрать жесткую минимальную ширину, отключить горизонтальную прокрутку и скорректировать sizing combo-box-ов.
  Date/Author: 2026-04-08, Codex.

- Decision: Для `v1.4.0` разбиение `Bounds` по зонам добавлено как core-фундамент, но `ensure_exportable()` продолжает блокировать область, пересекающую зоны.
  Rationale: Writer-ы KML/ZIP/SVG/GeoJSON/CSV еще не умеют выбирать трансформер по сегменту и рассчитывать глобальную нумерацию поверх нескольких сегментов. Хранение `GridStats.zone_segments` дает следующий шаг без риска молча преобразовать часть области через неверную зону.
  Date/Author: 2026-04-08, Codex.

- Decision: `split_bounds_by_zone()` режет область по внутренним миллионным границам `Y` и назначает зону сегмента по середине интервала.
  Rationale: `Bounds` в проекте работает как прямоугольный диапазон по ширине `y_right - y_left`; выбор точки внутри сегмента избегает ложной смены зоны на правой границе, которая одновременно является общей границей двух соседних сегментов.
  Date/Author: 2026-04-08, Codex.

- Decision: Для `v1.5.0` экспортируемая ячейка `1000x1000` или `100x100`, которая сама пересекает внутреннюю миллионную границу зоны, будет блокироваться понятной ошибкой; обычная многозонная область разрешается, если каждая экспортируемая ячейка целиком лежит в одной зоне или касается границы только ребром.
  Rationale: При выровненных сетках границы зон кратны и 1000 м, и 100 м, поэтому пользовательский сценарий из roadmap работает без разрезания квадратов и с глобальной нумерацией. В режиме без округления пользователь может задать смещенные границы, где один большой квадрат физически лежит в двух зонах; безопаснее запретить только этот частный случай, чем молча преобразовать часть полигона через неверный EPSG.
  Date/Author: 2026-04-08, Codex.

## Outcomes & Retrospective

Текущая задача roadmap-файла выполнена: создан корневой `ROADMAPv2.md`, который можно дать будущей сессии Codex или человеку без истории этого разговора. Реализация кода `v2.0` теперь начата: этап `v1.1.0` выполнен и принят пользователем как стабильная версия после ручного теста.

Итог `v1.1.0`: добавлен реестр экспортных форматов со стабильным `format_id`, SVG writer без новых зависимостей, UI-выбор SVG, сохранение `export_format_id` в `.lgm.json` рядом со старым `export_mode`, тесты и документация версии. ## Context and Orientation

LIMUZIN GRID MANAGER `v1.0.0` - Windows desktop-приложение на Python и PySide6 для генерации KML-сеток для AlpineQuest. Пользователь вводит две точки области в СК-42 / Пулково-42, Гаусса-Крюгера: `X` как северинг и `Y` как восток. Приложение строит сетки `1000x1000`, `100x100` или оба слоя вместе, преобразует координаты в WGS84 через `pyproj` и пишет KML.

Текущая важная координатная логика:

    zone = int(abs(Y) // 1_000_000)
    epsg = 28400 + zone
    lon, lat = transformer.transform(Y, X)

Поддерживаемые зоны в текущей ветке - `1..32`. Начиная с `v1.5.0`, область через несколько зон экспортируется автоматически, если каждая экспортируемая ячейка `1000x1000` или `100x100` целиком лежит в одной зоне или касается границы только ребром. Если в режиме без округления одна ячейка пересекает границу зоны внутри себя, экспорт блокируется понятной ошибкой до отдельной задачи по разрезанию таких ячеек.

Текущие ключевые файлы:

- `GRIDBASE.md` - главный технический источник по назначению приложения, координатам, KML, UI, ошибкам и правилам разработки.
- `roadmap.md` - исторический roadmap до `v1.0`; его не нужно заменять этим файлом.
- `PLANS.md` - правила ExecPlan-документов.
- `src/limuzin_grid_manager/core/models.py` - `Bounds`, `GridOptions`, `ExportMode`, `KmlStyle` и enum-настройки.
- `src/limuzin_grid_manager/core/crs.py` - определение зоны и создание трансформера СК-42 / Гаусса-Крюгера -> WGS84.
- `src/limuzin_grid_manager/core/zones.py` - определение зон, проверка диапазона `1..32` и разбиение `Bounds` на зональные сегменты.
- `src/limuzin_grid_manager/core/stats.py` - проверки экспортируемости, предупреждения и оценки размера.
- `src/limuzin_grid_manager/core/kml.py` - потоковая запись KML и ZIP.
- `src/limuzin_grid_manager/app/export_formats.py` - описания экспортных форматов, расширений, диалогов и сводки.
- `src/limuzin_grid_manager/app/exporter.py` - прикладной экспорт, проверка места, временный файл и атомарная замена результата.
- `src/limuzin_grid_manager/app/project.py` - сохранение и загрузка `.lgm.json`.
- `src/limuzin_grid_manager/ui/main_window.py` - основное окно, вкладки, диалоги и текущий встроенный stylesheet.
- `src/limuzin_grid_manager/ui/preview.py` - 2D-предпросмотр сетки без карты.

Термины этого roadmap:

SVG - векторный XML-формат для схемы сетки, которую можно открыть в редакторах векторной графики и использовать вне AlpineQuest.

GeoJSON - JSON-формат для GIS-инструментов. В этом проекте он должен хранить полигоны сетки в WGS84 и свойства ячеек: слой, зона, номер, имя и связь с большим квадратом.

CSV - табличный экспорт. В этом проекте он нужен для проверки и передачи номеров, имен, зон, границ в СК-42 и центров ячеек в WGS84.

Gauss-Kruger zone splitting / zone splitting - внутреннее разбиение области по зонам Гаусса-Крюгера, чтобы каждая часть сетки преобразовывалась в WGS84 через трансформер своей зоны.

### v1.2.0 - GeoJSON и CSV

Цель этапа - дать пользователю два практичных формата для GIS и табличной проверки без перегруза интерфейса и без CAD-сложности.

GeoJSON writer, ориентировочно `src/limuzin_grid_manager/core/geojson.py`, должен писать один `.geojson` как `FeatureCollection`. Геометрия каждой ячейки - `Polygon` в WGS84. Свойства feature должны включать как минимум `layer`, `zone`, `big_number`, `big_name`, `small_number`, а также признак, относится ли объект к `1000x1000` или `100x100`.

CSV writer, ориентировочно `src/limuzin_grid_manager/core/csv_export.py`, должен писать UTF-8 with BOM и разделитель `;`, чтобы файл комфортно открывался в русской локали Excel. В строках должны быть слой, зона, номер большого квадрата, пользовательское имя большого квадрата, номер малого квадрата, границы ячейки в СК-42 (`x_top`, `x_bottom`, `y_left`, `y_right`) и центр в WGS84 (`center_lon`, `center_lat`). CSV не заменяет `.lgm.json`: это экспорт результата, а не файл проекта.

DXF в этот этап и в обязательный объем `v2.0` не входит. Если DXF понадобится позже, для него нужен отдельный ExecPlan из-за CAD-единиц, слоев и возможных зависимостей.

Приемка этапа: добавлены тесты на валидный JSON GeoJSON, корректное количество features, координаты WGS84, свойства `zone` и номера; добавлены тесты на CSV-заголовки, `;`, BOM, строки для больших и малых квадратов, а также сводку и выбор файлов в UI.

### v1.4.0 - Фундамент автоматического разбиения по зонам

Цель этапа - безопасно подготовить многозонность в core до включения ее во все экспорты.

Нужно добавить модуль `src/limuzin_grid_manager/core/zones.py`. Он должен уметь определять зону по `Y`, проверять поддерживаемый диапазон `1..32` и разбивать `Bounds` на последовательность сегментов, где каждый сегмент целиком находится в одной зоне. Граница зон в текущей модели задается миллионами по `Y`, потому что зона считается как `int(abs(Y) // 1_000_000)`.

Сегмент зоны должен быть простой структурой, например dataclass с `zone: int` и `bounds: Bounds`. Конкретное имя можно выбрать при реализации, но его назначение должно быть очевидным: один сегмент равен одной части исходной области, которую можно трансформировать одним EPSG `28400 + zone`.

На этом этапе статистика может все еще блокировать многозонный экспорт, если экспортные писатели не готовы. Но в `core/stats.py` нужно подготовить модель сообщений так, чтобы будущий этап мог заменить жесткую ошибку на сводку сегментов. Однозонное поведение обязано остаться прежним.

Приемка этапа: unit-тесты покрывают область внутри одной зоны, область на границе зон, область через две зоны, область через несколько зон, зоны вне `1..32`, а также сохранение прежней ошибки там, где экспорт еще не включил многозонность.

### v1.5.0 - Многозонный экспорт KML, SVG, GeoJSON и CSV

Цель этапа - включить автоматическое разбиение по зонам как пользовательскую функцию.

При экспорте области, пересекающей зоны, приложение должно разделить работу на zone segments и использовать правильный трансформер для каждого сегмента. Для пользователя это должно оставаться одним действием: он задает одну область, выбирает формат и нажимает экспорт.

KML, SVG, GeoJSON и CSV должны поддерживать многозонность. ZIP также должен сохранять совместимость: имена файлов внутри архива остаются `tile_###.kml`. Если большой квадрат целиком находится в одной зоне, он пишется как раньше, но с трансформером своей зоны. Если при реализации выяснится, что большой квадрат может быть разрезан границей зоны, нужно зафиксировать решение в `Decision Log` этого файла и выбрать безопасное поведение до изменения кода: либо делить такой квадрат на зональные части с понятными именами, либо запрещать только этот частный случай с ясной ошибкой. Нельзя молча трансформировать часть квадрата через неправильную зону.

Нумерация больших квадратов должна оставаться глобальной по всей области, чтобы переход к многозонности не менял привычные номера пользователя. Нумерация малых `100x100` остается локальной внутри каждого `1000x1000`, как в `v1.0.0`. Переименование больших квадратов не должно менять номера малых квадратов.

Приемка этапа: тесты доказывают, что прежняя зона-crossing ошибка заменена на успешный экспорт там, где это поддержано; KML и GeoJSON используют координаты WGS84 из правильных зон; SVG и CSV содержат `zone`; ZIP сохраняет `tile_###.kml`; прогресс, отмена и атомарная запись через временный файл продолжают работать.

Этот файл является добавлением к репозиторию. Он не заменяет `roadmap.md` и не требует менять код, тесты, версии или сборочные файлы только ради своего существования.

Если будущий агент начинает работу и `ROADMAPv2.md` уже существует, он должен сначала прочитать этот файл и обновить его как living document. Нельзя слепо перезаписывать `Progress`, `Surprises & Discoveries`, `Decision Log` или `Outcomes & Retrospective`.

Если во время реализации выяснится, что выбранная архитектура экспорта или многозонности не подходит, нужно записать наблюдение в `Surprises & Discoveries`, добавить решение в `Decision Log`, затем обновить соответствующий milestone. Изменения курса должны оставаться понятными будущему читателю без истории чата.

Любое изменение, которое может сломать старые `.lgm.json`, KML или ZIP сценарии, должно иметь миграционный или совместимый путь. Если совместимость невозможна, это должно быть явно записано как major-version решение до изменения кода.

## Interfaces and Dependencies

Планируемые новые или переработанные модули:

- `src/limuzin_grid_manager/core/zones.py`: разбиение `Bounds` по зонам, проверка поддерживаемого диапазона зон и zone-aware helpers.
- `src/limuzin_grid_manager/core/svg.py`: запись SVG без новой зависимости.
- `src/limuzin_grid_manager/core/geojson.py`: запись GeoJSON через стандартный модуль Python `json`.
- `src/limuzin_grid_manager/core/csv_export.py`: запись CSV через стандартный модуль Python `csv`.
- `src/limuzin_grid_manager/ui/themes.py`: имена тем, QSS и функции применения темы.

Новые зависимости не требуются для SVG, GeoJSON и CSV. SVG можно писать как XML/text. GeoJSON можно писать через `json`. CSV можно писать через `csv`.

Если позже понадобится DXF или PDF, нужно создать отдельный ExecPlan до добавления зависимости, потому что такие форматы требуют отдельных решений по координатам, слоям, единицам, шрифтам и совместимости.
